# SPTronic Support System — описание функционала для пользователей

## Для чего нужна система

SPTronic Support System нужна как единая площадка для работы с обращениями:

- пользователь создает тикет и отслеживает решение;
- саппорт обрабатывает тикеты и ведет коммуникацию;
- админ контролирует общую картину и пользователей;
- при необходимости подключаются временные специалисты.

Главная ценность системы — прозрачный и понятный процесс решения проблем: от создания обращения до закрытия с оценкой качества.

---

## Как пользователь попадает в систему

Работа начинается со страницы входа/регистрации.  
После входа человек попадает в интерфейс в соответствии со своей ролью.  
Роль определяет:

- какие страницы доступны;
- какие действия можно выполнять;
- какие данные видны.

Это нужно, чтобы каждый участник видел только нужный ему функционал и не перегружался лишним.

---

## Роли и функционал

## 1) Пользователь

Пользователь — это человек, который обращается в поддержку.

### Что ему доступно

- Личный кабинет:
  - просмотр своего профиля;
  - редактирование ФИО, логина, пароля;
  - просмотр ID и email (без изменения);
  - просмотр доступных модулей.
- Список своих тикетов:
  - таблица тикетов с ключевыми полями;
  - сортировка по колонкам.
- Создание нового тикета:
  - выбор модуля;
  - описание проблемы;
  - прикрепление файлов.
- Карточка тикета:
  - просмотр описания и статуса;
  - чат с саппортом;
  - загрузка пользовательских файлов;
  - просмотр лог-файлов от саппорта;
  - отправка жалобы.
- После закрытия:
  - оценка работы саппорта по 5-балльной шкале;
  - необязательный комментарий;
  - возможность отложить оценку.

### Зачем это нужно

Пользователь всегда понимает, что происходит с его обращением, может оперативно дополнять информацию и влиять на качество поддержки.

---

## 2) Саппорт

Саппорт — это исполнитель, который решает обращения.

### Что ему доступно

- Раздел "Все тикеты":
  - общая очередь обращений;
  - фильтрация по статусу;
  - поиск;
  - сортировка по колонкам;
  - взятие тикета в работу.
- Раздел "Мои тикеты":
  - список закрытых тикетов для личной отчетности.
- Карточка тикета:
  - чат с пользователем;
  - прикрепление лог-файлов;
  - закрытие тикета;
  - временная блокировка пользователя (до 12 часов) с обязательной причиной.

### Зачем это нужно

Саппорт быстро ориентируется в очереди, качественно ведет коммуникацию и фиксирует все действия в одном месте.

---

## 3) Временный саппорт (сторонний автор)

Временный саппорт — внешний участник, подключаемый точечно.

### Что ему доступно

- Только один конкретный тикет, в который его пригласили.
- Работа в рамках этого тикета: просмотр, обсуждение, помощь в решении.

### Зачем это нужно

Можно безопасно подключать внешнего эксперта без открытия доступа ко всей системе.

---

## 4) Админ

Админ — роль контроля и управления.

### Что ему доступно

- Дашборд:
  - общая статистика по тикетам;
  - текущая нагрузка;
  - распределение по статусам.
- Тикеты:
  - просмотр всех тикетов;
  - сортировка/фильтрация;
  - контроль процесса.
- Пользователи:
  - страница "Все пользователи";
  - страница "Пользователи" (только роль пользователь);
  - страница "Саппорты" (только роль саппорт).

### Зачем это нужно

Админ видит текущее состояние поддержки, может контролировать качество процессов и структуру пользователей.

---

## 5) Автор-разработчик

Роль автора-разработчика по доступам приравнена к администратору.

### Что ему нужно

- Доступ ко всем тикетам;
- Возможность подключаться к сложным кейсам;
- Контроль ситуации на уровне всей системы.

### Зачем это нужно

Ускоряет решение нетипичных технических проблем и сокращает путь между поддержкой и разработкой.

---

## Ключевые страницы и смысл каждой

## Страница входа/регистрации

Нужна, чтобы пользователь начал работу и попал в интерфейс в рамках своей роли.

## Личный кабинет

Нужен для управления персональными данными и понимания, какие модули доступны конкретному пользователю.

## Новый тикет

Нужен для корректного старта обращения: модуль + описание + файлы.  
Чем качественнее заполнен тикет, тем быстрее решение.

## Список тикетов

Нужен как рабочий реестр обращений.  
По таблице должно быть легко:

- найти нужный тикет;
- отсортировать;
- открыть карточку.

## Карточка тикета

Это центральная рабочая зона: здесь собрана вся история по проблеме:

- исходное описание;
- чат;
- файлы;
- лог-файлы;
- статус;
- ключевые действия по роли.

## Админский дашборд

Нужен для оперативного контроля состояния сервиса поддержки.

## Страницы пользователей (админ)

Нужны для просмотра состава участников системы и разделения на клиентские и саппорт-аккаунты.

---

## Как должен выглядеть "идеальный" процесс

1. Пользователь создает тикет и описывает проблему.
2. Саппорт видит тикет в очереди и берет его в работу.
3. В карточке ведется переписка и обмен файлами.
4. Саппорт устраняет проблему и закрывает тикет.
5. Пользователь оценивает качество поддержки.
6. Админ видит результат в статистике и контролирует общую эффективность.

---

## Что важно для удобства людей

- Единый визуальный формат таблиц на всех ролях.
- Понятные статусы тикетов.
- Трехсостоящая сортировка по колонкам:
  - по умолчанию;
  - по возрастанию;
  - по убыванию.
- Прозрачный доступ по ролям: каждый видит только то, что относится к его задачам.

---

## Итог

Разметка и текущий UI-прототип реализованы для демонстрации полного пользовательского пути:

- от входа в систему;
- до создания и сопровождения тикета;
- до закрытия и оценки качества поддержки;
- с управленческим контролем на стороне админа.

Это позволяет согласовать процесс с бизнесом и командами до подключения backend-логики.
